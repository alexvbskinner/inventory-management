System Architecture

Factory Inventory Management System

A full-stack demo application for inventory management, order tracking, demand forecasting, and spending analytics across factory operations. Vue 3 single-page frontend, Python FastAPI backend, and in-memory mock data loaded from JSON.

Vue 3 SPA FastAPI REST / JSON In-memory data Bilingual (EN / JA)

System Architecture

A classic three-layer architecture. A Vite-served Vue SPA in the browser talks to a stateless FastAPI service over HTTP/REST; the API reads from datasets held in memory, hydrated from JSON files at process startup.

Stateless & ephemeral. Data is loaded into Python lists/dicts at startup and never written back — changes do not persist, and restarting the server reloads the original JSON. There is no database, authentication, or persistence layer; the system is intended for local demonstration only.

Data Flow — A Filtered Request

The four global filters (Time Period, Warehouse, Category, Order Status) drive nearly every screen. Here is the round trip when a user changes a filter on the dashboard.

User adjusts a filter

A selection in FilterBar.vue updates shared reactive refs in the useFilters composable (e.g. selectedWarehouse).

View reacts and calls the API

A view such as Dashboard.vue watches the filters and invokes a method on api.js, e.g. api.getDashboardSummary(filters).

axios builds the request

api.js serializes active filters into query params (skipping any set to 'all') and issues a GET to http://localhost:8001/api/....

FastAPI filters in memory

The route handler in main.py passes the in-memory dataset through apply_filters() and filter_by_month() (month or Q1–Q4 quarter matching on order_date).

Pydantic validates the response

Results are returned through a response_model (e.g. List[Order]), guaranteeing shape and types before serialization to JSON.

Vue renders derived state

The view stores raw results in refs (allOrders, inventoryItems) and exposes computed properties for metrics, charts (custom SVG), and tables.
